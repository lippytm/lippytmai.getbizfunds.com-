# AI Self-Healing Review Checklist

## Purpose
Use this checklist when reviewing whether a repo, CRM surface, workflow, or shared asset set needs self-healing actions.

## Visibility Review
- [ ] repo role is visible
- [ ] purpose is visible
- [ ] status is current enough
- [ ] roadmap is visible enough
- [ ] next action is visible

## Continuity Review
- [ ] lifecycle stage is current where relevant
- [ ] handoff summary is strong enough
- [ ] ownership is visible
- [ ] dependencies are visible
- [ ] follow-on action is clear

## Reuse Review
- [ ] reusable assets are surfaced clearly
- [ ] prompts are indexed clearly
- [ ] duplicated assets are not silently drifting
- [ ] canonical ownership is visible

## Risk Review
- [ ] trust risk is visible where relevant
- [ ] support risk is visible where relevant
- [ ] churn risk is visible where relevant
- [ ] innovation maturity is visible where relevant

## Healing Outcome
- [ ] visibility repair needed
- [ ] continuity repair needed
- [ ] ownership repair needed
- [ ] reuse repair needed
- [ ] promotion repair needed

## Best Practices
- heal the smallest highest-leverage issue first
- document why healing is needed
- use this checklist to improve scanability and continuity before deeper changes
