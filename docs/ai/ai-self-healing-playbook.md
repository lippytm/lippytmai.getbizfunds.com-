# AI Self-Healing Playbook

## Purpose

This playbook defines practical self-healing patterns for restoring healthier AI ecosystem states when documentation, CRM continuity, status visibility, ownership clarity, or reuse surfaces weaken.

## Self-Healing Priorities

### Priority 1 — Restore Visibility
Examples:
- missing status file
- missing next action
- stale roadmap
- unclear repo role

### Priority 2 — Restore Continuity
Examples:
- missing CRM next-best action
- stale lifecycle stage
- weak handoff summary
- missing support resolution note

### Priority 3 — Restore Ownership Clarity
Examples:
- duplicated standards without canonical owner
- unclear asset source
- unclear prompt ownership

### Priority 4 — Restore Reuse Surfaces
Examples:
- hidden prompt inventory
- hidden asset inventory
- missing workflow inventory
- missing learning surface

### Priority 5 — Promote Learning
Examples:
- lessons stranded in innovation repos
- useful templates not moved into knowledge repos
- durable CRM patterns not documented well enough

## Healing Actions by Type

### Action Type A — Visibility Repair
Add or refresh:
- STATUS.md
- ROADMAP.md
- FLEET.md
- next action notes

### Action Type B — Context Repair
Add or refresh:
- summaries
- handoff notes
- integration references
- lifecycle fields

### Action Type C — Ownership Repair
Add or refresh:
- ASSETS.md
- canonical ownership note
- registry entry
- source reference

### Action Type D — Reuse Repair
Add or refresh:
- prompt inventory
- workflow inventory
- learning inventory
- shared asset links

### Action Type E — Promotion Repair
Add or refresh:
- promotion decision
- learning summary
- stronger target repo placement
- knowledge or standards reference

## Best Practices
- repair the smallest highest-leverage gap first
- restore scanability before deeper refactoring
- document why the repair matters
- prefer reusable fixes over one-off patches
- keep human review for high-impact standards changes when needed
