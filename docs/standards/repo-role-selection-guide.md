# Repo Role Selection Guide

## Purpose
Use this guide to decide the main role a repository should play in the wider ecosystem.

## Repo Roles

### front_door
Mainly public-facing. Helps route people into services, knowledge, products, or relationship paths.

### brand
Mainly owns brand identity, umbrella positioning, or top-level direction.

### workflow
Mainly owns automation, orchestration, or internal process support.

### product
Mainly owns a service, implementation package, application, or productized capability.

### knowledge
Mainly owns reusable learning, SOPs, prompts, standards, or playbooks.

### innovation
Mainly owns experiments, prototypes, sandbox ideas, or rapid testing.

### control_tower
Mainly owns visibility, status rollups, governance, or fleet coordination.

## Selection Questions
Ask:
1. Is this repo mainly public-facing or mainly specialized?
2. Does it mainly present, operate, teach, build, test, or coordinate?
3. Which other repos depend on it most?
4. What is its strongest capability today?

## Best Practices
- choose one main role even if a repo has several strengths
- record secondary strengths in CAPABILITIES.md
- keep the role stable unless the repo changes a lot over time
