# Standards Application Sequence

## Purpose

This guide recommends the order for applying the standards, fleet application pack, and fleet-ready repo package set across repositories.

The goal is to maximize ecosystem leverage early instead of spreading effort too thin.

---

## Priority Principle

Upgrade the repos that improve visibility, routing, reuse, and orchestration first.

That usually means:
1. front-door repo
2. brand repo
3. workflow repo
4. product repo
5. knowledge repo
6. control-tower repo
7. innovation repos

---

## Recommended Rollout Phases

### Phase 1 — Front-Door Foundation
Target repos:
- lippytmai.getbizfunds.com-
- other public-facing entry repos

Why first:
- public clarity
- relationship routing
- CRM entry consistency
- ecosystem navigation

Required files:
- README
- CAPABILITIES
- INTEGRATIONS
- STATUS
- ROADMAP
- STANDARDS
- PROMPTS
- ASSETS

### Phase 2 — Brand and Core Identity
Target repos:
- lippytm.ai
- top-level identity or umbrella repos

Why second:
- shared positioning
- brand coherence
- architecture alignment

### Phase 3 — Workflow and Operations
Target repos:
- MyClaw-related repos
- automation and orchestration repos

Why third:
- operational leverage
- reusable logic
- routing and execution support

### Phase 4 — Product and Offer Repos
Target repos:
- BotBuilders-related repos
- service and product repos

Why fourth:
- offer alignment
- implementation packaging
- better front-door to product handoff

### Phase 5 — Knowledge and Learning Repos
Target repos:
- SOP repos
- prompt repos
- documentation hubs

Why fifth:
- shared assets become easier to reference and reuse

### Phase 6 — Control Tower
Target repos:
- orchestration hub
- fleet visibility repo

Why sixth:
- by this stage more repos can report cleanly into rollups

### Phase 7 — Innovation and Sandbox Repos
Target repos:
- experiment repos
- prototype repos

Why seventh:
- keep experiments flexible while still bringing enough standards to remain understandable

---

## Best Practices
- do not try to normalize every repo at once
- start where shared clarity creates the most leverage
- use examples, not only blank templates
- audit each repo before deciding how deeply to upgrade it
- preserve specialization while standardizing visibility
